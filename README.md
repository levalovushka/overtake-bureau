# Overtake — лендинг бюро

Один экран, без скролла (на десктопе): хиро, слайдер кейсов, питч, CTA. Никакого фреймворка и сборки — чистые `index.html` / `style.css` / `app.js`.

## Стек и структура

```
index.html    — разметка, вся она на одну секцию (английская версия, корень сайта)
ru/index.html — русская версия той же страницы (см. «Локализация» ниже)
_redirects    — правила Netlify: авторедирект `/` → `/ru/` по языку браузера / стране (см. «Локализация»)
style.css     — вся стилистика, один файл, общий для обеих языковых версий
app.js        — вся интерактивность (слайдер), один IIFE, без зависимостей, общий для обеих версий
assets/       — .webp картинки и .mp4 видео кейсов
assets/icons/ — исходники SVG-иконок (сейчас: стрелки и drag-иконка курсорной кнопки слайдера)
fonts/        — Pragmatica Next, статические subset'ы под вес 400: латиница + кириллица (см. «Шрифт» ниже)
fonts/source/ — оригинальный вариативный шрифт, в .gitignore (не в репо, нужен только чтобы пересобрать subset)
```

Зависимостей нет вообще — ни `package.json`, ни бандлера, ни постпроцессора CSS. Это осознанный выбор (см. «Сверхминимализм» ниже), а не недоделанность. Прежде чем предлагать webpack/vite/postcss/tailwind — спроси себя, действительно ли сложность страницы это оправдывает.

`skills-lock.json` и `.agents/`/`.claude/` — это не часть сайта, а конфиг для Claude Code (см. «Скилл для анимации» ниже). `.agents/`/`.claude/` в `.gitignore`, `skills-lock.json` закоммичен, чтобы `npx skills add` можно было переустановить в той же версии.

## Деплой

- Репозиторий: [github.com/levalovushka/overtake-bureau](https://github.com/levalovushka/overtake-bureau)
- Подключён к **Netlify** напрямую от репо: любой пуш в `main` — автодеплой, без ручной заливки.
- Билд-шага нет и не нужен (в Netlify это должно быть настроено как publish directory = корень репо, build command — пустой). Если Netlify вдруг начнёт просить build command — значит, кто-то по ошибке завёл там фреймворк-пресет, чинить надо именно в настройках Netlify, не добавлением сборки в проект.
- Значит: `git push` в `main` = сайт обновился. Тестировать перед пушем — локально (`python3 -m http.server` или любой статик-сервер), staging-окружения нет.

## Принципы

Три вещи, которыми стоит руководствоваться при любой правке.

### Сверхминимализм

- Один текстовый стиль на весь сайт (`--font-size`/`--line-height`/`--font-weight` в `:root`, используются буквально везде — не заводи вторую типографику без крайней необходимости).
- Одна колонка контента (`--content-width`), к ней прибиты хиро, питч, CTA и (с недавних правок) левый край текущего кейса в слайдере — так столбец текста и картинка всегда на одной вертикали.
- Одна секция, один разворот экрана — вся вертикальная раскладка живёт в `--space-page-top`/`--case-height`, посчитанных через `clamp()` от высоты вьюпорта, а не через media-запросы на каждый чих.
- Никаких абстракций про запас: нет компонентной системы, нет CSS-методологии — файл один, он маленький, он весь читается за минуту. Не разбивай его на модули, пока он не перестал в неё помещаться.

### Кайф в деталях

Сайт держится на мелочах, которые пользователь не заметит осознанно, но почувствует:
- Кастомный курсор-кнопка на слайдере (десктоп): вместо системного курсора — круглая стеклянная кнопка, которая показывает ← или →, смотря на какую половину секции навёл; смена иконки — блюр + скейл-даун + опасити, а не мгновенный свап.
- «Peek» — кейс никогда не занимает весь экран впритык, всегда видно кусок соседней карточки, чтобы было понятно, что можно листать дальше (свайпом на мобиле, кликом/наведением на десктопе).
- Кастомные easing-кривые, а не дефолтные CSS: `--ease-out: cubic-bezier(0.23, 1, 0.32, 1)` — сильный ease-out, а не встроенный `ease-out`, который слишком вялый.
- `prefers-reduced-motion` уважается (см. `@media (prefers-reduced-motion: reduce)` в конце `style.css` и `reduceMotion` в `app.js`) — это тоже деталь, а не галочка для галочки.
- `corner-shape: squircle` на рамке кейса — сквирк, а не обычный border-radius, там где браузер это поддерживает (прогрессивное улучшение, деградирует в обычное скругление).

Если добавляешь новую интерактивность — задай себе вопрос из скилла ниже: «эта анимация вообще нужна, и если да — на чём именно должна быть ease-out, а на чём ease-in-out».

### Сверхоптимизация

- Ни одной лишней зависимости в рантайме — весь JS это один файл без библиотек.
- Анимируются только `transform`/`opacity`/`filter` — то, что не триггерит layout/paint.
- Шрифт — статический subset под единственный реально используемый вес (400), а не вариативный на весь диапазон 100–900: 772 KB → 32 KB. `font-display: swap`, шрифт — `preload`. Подробности и команда пересборки — в разделе «Шрифт» ниже.
- Картинки — `.webp`, `loading`/`fetchpriority` расставлены по тому, что реально видно в первом кадре слайдера (см. «Медиа кейсов» ниже), а не по порядку в массиве.
- Слайдер клонирует ровно столько соседних слайдов, сколько реально может быть видно при текущей ширине экрана (`neededBuffer` в `app.js`), а не фиксированное число — меньше лишних DOM-узлов и картинок. Клоны всегда вне экрана в момент первой отрисовки (это просто запас для зацикливания), поэтому `makeClone()` в `app.js` принудительно ставит им `loading="lazy"` — не отнимают полосу у того, что реально видно.

Правки, которые тянут за собой сборщик, лишние килобайты JS или лишние DOM-узлы «на всякий случай», — по умолчанию не в духе проекта. Если без этого правда не обойтись — ок, но осознанно.

### Шрифт

`fonts/pragmatica-next-400.woff2` — не вариативный шрифт, а статический инстанс, вытянутый из оригинального variable-файла и засабсеченный под нужные символы. На сайте используется только один вес (400) и одна ширина (100%, `wdth`-ось не используется нигде в CSS) — держать все 800 градаций веса и весь диапазон растяжения ради одного используемого значения бессмысленно.

Оригинальный вариативный шрифт лежит в `fonts/source/pragmatica-next_vf.woff2` — эта директория в `.gitignore` и **не попадает в репозиторий**. Она нужна только на машине, где пересобирается subset (например, если в будущем понадобится другой вес — какой-нибудь `font-weight: 700` для акцента). Если `fonts/source/` отсутствует, а вариативный шрифт нужен заново — попроси оригинальный файл у того, кто его изначально прислал (шрифт покупной/лицензионный, это не то, что можно просто перекачать).

Команда пересборки (нужен `fontTools`, `pip install fonttools`):

```bash
# 1. вытянуть статический инстанс на нужном весе/ширине из вариативного шрифта
python3 -m fontTools.varLib.instancer \
  fonts/source/pragmatica-next_vf.woff2 wght=400 wdth=100 \
  -o /tmp/static-400.woff2

# 2. засабсетить под реально используемые символы (латиница + типографские тире/кавычки/многоточие)
python3 -m fontTools.subset /tmp/static-400.woff2 \
  --unicodes="U+0020-007E,U+00A0,U+2013,U+2014,U+2018-201B,U+2026" \
  --layout-features='*' \
  --flavor=woff2 \
  --no-hinting \
  --output-file=fonts/pragmatica-next-400.woff2

# 3. отдельный кириллический subset для /ru/ (А-я, Ё/ё, «», ₽)
python3 -m fontTools.subset /tmp/static-400.woff2 \
  --unicodes="U+00AB,U+00BB,U+0401,U+0410-044F,U+0451,U+20BD" \
  --layout-features='*' \
  --flavor=woff2 \
  --no-hinting \
  --output-file=fonts/pragmatica-next-400-cyrillic.woff2
```

Кириллица лежит отдельным файлом (`pragmatica-next-400-cyrillic.woff2`, ~12 KB) со своим `@font-face` + `unicode-range` в `style.css`: браузер скачивает его только на страницах, где кириллица реально встречается (то есть на `/ru/`), английская версия по-прежнему тянет только латинский файл. На `/ru/` оба файла в `preload` (латиница там тоже есть — «Overtake», цифры).

Если в тексте на сайте появится символ вне этих диапазонов (новые типографские знаки, другой алфавит) — subset его не будет содержать, и браузер молча упадёт на `system-ui` для этого символа. В таком случае просто расширь `--unicodes` нужного subset'а **и его `unicode-range` в `style.css`** и пересобери.

## Локализация

Две языковые версии, **без переключателя языка на сайте** — это осознанное решение, не забытая фича:

- `/` — английская (`index.html`), `/ru/` — русская (`ru/index.html`). CSS/JS/картинки общие, поэтому все пути в обоих HTML — абсолютные (`/style.css`, `/assets/…`); относительные из `ru/` указывали бы не туда.
- Кто попадает на русскую версию, решает Netlify на CDN — правила в `_redirects`: с `/` на `/ru/` редиректятся браузеры с русским первым языком (`Language=ru,ru-ru`) и посетители из РФ по GeoIP (`Country=ru`). Редирект **только с корня** и только `302` — это рекомендация Google для авторедиректа по локали: `/ru/` всегда доступна напрямую, краулеры видят обе версии. Не добавляй обратный редирект с `/ru/` на `/` и не меняй 302 на 301.
- Обе страницы связаны `hreflang`-ссылками (`en` / `ru` / `x-default` → английская) с абсолютными URL на `https://overtakers.netlify.app`. Если сайт переедет на кастомный домен — эти URL надо обновить в обоих HTML.
- Ограничение, о котором стоит помнить: Netlify матчит только **первый** язык из `Accept-Language` (q-веса игнорирует), так что русскоязычный человек с англоязычной системой и русским вторым языком получит английскую версию, если он не в РФ. Это принятый компромисс. Аварийный люк на будущее: куки `nf_lang` / `nf_country` переопределяют детект Netlify — переключатель можно сделать без изменения правил.
- Правки текста вносить в **обе** версии руками — никакой автоматической синхронизации нет. Русская типографика: `&nbsp;` после коротких предлогов/союзов и перед `—`, цена — `800&nbsp;000&nbsp;₽`.
- Локально редиректы не проверить (`python3 -m http.server` про `_redirects` не знает) — только на проде: `curl -sI -H "Accept-Language: ru" https://overtakers.netlify.app/` должен отдать `302` с `location: /ru/`.

## Слайдер кейсов — как устроен

Вся логика в `app.js`, без сторонних либ. Ниже — полная модель: её должно хватать, чтобы доводить ощущения или добавлять визуальные эффекты, не восстанавливая архитектуру из кода.

### Модель состояния

Всё движение сводится к одному непрерывному числу:

- `pos` — текущая позиция в единицах слайдов: `2.0` — третий кейс выровнен по колонке, `2.5` — ровно между третьим и четвёртым. Драг пишет в `pos` напрямую, спринг интегрирует его к `target`.
- `target` — куда едем; всегда целое (снэп-точка). Клики, клавиатура и флик меняют только его.
- `velocity` — скорость в слайдах/сек. При отпускании пальца сюда записывается скорость жеста, и спринг стартует с неё — поэтому между драгом и доводкой нет шва (velocity handoff).
- `drag` — состояние текущего жеста или `null` (см. «Жесты»).
- Геометрия кэшируется в `buildLoop()` и пересчитывается только на resize: `itemStep` (ширина слайда + gap), `colLeft` (левый край текстовой колонки относительно секции, меряется по `.hero`), `bufferCount` (сколько клонов с каждой стороны).

Единственное место, где `pos` превращается в пиксели — `render()`:

```
x = colLeft − (bufferCount + pos) · itemStep
track.style.transform = translate3d(x, 0, 0)
```

Левый край текущего кейса всегда совпадает с левым краем текстовой колонки, а не центрируется в вьюпорте — так на мобиле видно, что справа есть что свайпнуть, а слева ничего не «висит». «Текущий слайд» в любой момент = `mod(Math.round(target), count)`.

### Кадровый цикл (tick)

rAF-цикл живёт только пока есть движение (спринг не доехал или идёт драг); в покое не выполняется ни одного кадра.

- Спринг критически демпфированный: `v += (−ω²·(pos−target) − 2ω·v)·dt`, `pos += v·dt`. Осцилляций нет по построению; перелёт с возвратом возможен только если у жеста была реальная скорость — это осознанно: овершут «из физики», а не скриптованный bounce. Нет ни фиксированной длительности, ни флага `isAnimating` — перенацеливание на любом кадре легально.
- Интеграция идёт сабстепами ≤8мс: один большой шаг (пропущенный кадр или возврат из фоновой вкладки; dt клампится в 64мс) выходит за зону устойчивости полу-неявного Эйлера и взрывает спринг — сабстеппинг не убирать. При `SPRING_OMEGA` выше ~60 сабстеп нужно уменьшать.
- Покой: `|pos−target| < 0.001` и `|v| < 0.01` → снэп в точное значение, цикл останавливается.
- Во время закоммиченного драга физика на паузе — цикл просто рендерит `pos`, который пишет pointermove.

### Бесконечный луп

Реальных кейсов 5; `buildLoop()` достраивает с каждой стороны ровно столько клонов, сколько слайдов может выглядывать в вьюпорт при текущей ширине (ни одного «пустого» кадра). `wrap()` перебазирует `pos`, `target` и базу драга на целое число кругов — на любом кадре, в том числе посреди драга и полёта. Сдвиг ровно на `count` слайдов визуально неотличим (по краям стоят точные копии), поэтому тащить можно бесконечно в любую сторону, клампов по границе буфера нет.

### Жесты (один обработчик на мышь и тач, pointer events)

`pointerdown` создаёт «незакоммиченный» жест — кандидат в клик; спринг при этом **не** останавливается.

1. **Коммит**: движение ≥ `DRAG_THRESHOLD` (6px) по X. На тач — если |dy| > |dx|, жест отдаётся нативному вертикальному скроллу (`touch-action: pan-y` в CSS). В момент коммита база пересчитывается (иначе трек прыгнул бы на величину порога), спринг замирает, указатель захватывается через `setPointerCapture`. Схваченный на лету трек ловится там, где визуально находится.
2. **Ведение**: `pos = basePos + (baseX − clientX) / itemStep` — 1:1 за указателем. История последних ~100мс позиций копится в `drag.samples` для замера скорости.
3. **Отпускание** — цепочка правил снэпа, по порядку:
   - скорость релиза считается по сэмплам; пауза перед отпусканием > `RELEASE_STALL` (80мс) обнуляет её («положил на место»);
   - проекция импульса: `next = round(pos + v · PROJECTION)` — ближайший слайд к точке, куда жест «докатился бы» (экспоненциальное затухание, модель iOS-скролла). Сильный флик уезжает на 2–4 слайда;
   - гарантия: медленный драг дальше `SNAP_THRESHOLD_RATIO` (15% ширины) без импульса всё равно двигает на 1;
   - кламп: не дальше `MAX_FLICK` (= count) слайдов за один жест;
   - `velocity = скорость релиза`, `target = next` — дальше доводит спринг.
4. **Клик** (жест так и не закоммитился, мышь): навигация по кликнутой половине секции, `goTo(±1)`.

`goTo` накапливает на `target`, а не на `pos` — быстрые повторные клики складываются (два клика = два слайда). Никакого `isAnimating`-замка нет, и возвращать его нельзя: вся отзывчивость построена на его отсутствии.

Стрелки ←/→ работают при фокусе на секции (`tabindex="0"`). Краевые случаи закрыты: правая кнопка мыши игнорируется (контекстное меню съедает pointerup и подвесило бы жест), второй палец во время драга игнорируется, незакоммиченный жест, уехавший за секцию, сбрасывается на pointerleave, resize посреди жеста роняет драг и мгновенно ставит трек на целевой слайд.

### Ручки настройки ощущений

Все — именованные константы в начале `app.js`:

| Константа | Сейчас | Что крутит |
| --- | --- | --- |
| `SPRING_OMEGA` | 16 | Резвость доводки. Выше — жёстче и быстрее (20 ≈ 350мс до визуального покоя), ниже — плавнее и ленивее (12 ≈ 700мс). |
| `PROJECTION` (decay внутри) | 0.998 | Дальность флика. 0.998 — щедрый iOS-скролл (импульс ≈ v·0.5с), 0.997 — короче и собраннее (≈ v·0.33с). |
| `SNAP_THRESHOLD_RATIO` | 0.15 | Доля ширины слайда, после которой медленный драг «считается» перелистыванием. |
| `DRAG_THRESHOLD` | 6px | Порог, отделяющий клик от драга. |
| `VELOCITY_WINDOW` | 100мс | Окно замера скорости релиза. Меньше — нервнее реагирует на последний рывок пальца. |
| `RELEASE_STALL` | 80мс | Сколько нужно «зависнуть» перед отпусканием, чтобы импульс обнулился. |
| `MAX_FLICK` | count | Потолок слайдов за один жест. |

`prefers-reduced-motion` слушается на лету: все перемещения становятся мгновенными (`retarget` снэпает без спринга), видео в кейсах останавливаются. Драг 1:1 остаётся — это прямое манипулирование, а не анимация.

### Куда вешать визуальные эффекты

- Единственная точка «позиция → DOM» — `render()`. Эффекты, зависящие от позиции (параллакс, масштаб/затемнение соседних, прогресс-индикатор), считать там: для элемента `items[j]` расстояние до точки выравнивания = `j − bufferCount − pos` (0 — слайд на месте, ±1 — соседи). Эффект обязан быть чистой функцией `pos`: `wrap()` перебазирует значения на целые круги, и любое состояние, накопленное между кадрами в абсолютных величинах, сломается.
- Для эффектов от скорости (skew, motion-blur) есть `velocity` (слайды/сек) — но во время драга он заморожен; внутри драга скорость при желании считать по дельте `pos` между кадрами tick.
- В tick/`render()` — только compositor-friendly свойства (`transform`, `opacity`, `filter`) и никакого чтения layout (`getBoundingClientRect` и т.п.): геометрия уже закэширована, форс-рефлоу на каждом кадре убьёт плавность.
- Эффекты должны одинаково доставаться клонам (`[data-clone="true"]`) — по краям вьюпорта видны именно они. Клоны создаются копированием реальных слайдов в `buildLoop()` (на старте и на каждый resize) — стили/классы, навешанные на реальные слайды до этого, скопируются сами; всё, что вешается позже, нужно вешать и на клоны.
- Момент «доехали до слайда N» (для дотов-индикаторов и т.п.): событий в коде нет — ловить смену `mod(round(target), count)` в `retarget()`/`goTo()` или момент покоя в tick.

### Прочее

- Иконки курсорной кнопки (`prev`/`next`/`drag`) — исходники лежат в `assets/icons/*.svg`, но в разметке заинлайнены прямо в `index.html`, с заменой `fill="#101010"` на `fill="currentColor"`. Так нужно, чтобы иконка наследовала цвет кнопки (белый на чёрном фоне) и участвовала в blur/scale/opacity-кроссфейде между состояниями — оба завязаны на CSS, `<img src="assets/icons/...">` этого не даёт. Если добавляешь новую иконку в кнопку — тот же путь: исходник в `assets/icons/`, инлайн в HTML с `currentColor`.
- Десктоп-курсор (`hover: hover` и `pointer: fine`): при наведении системный курсор прячется, вместо него следует кастомная кнопка с иконкой; на время драга иконка меняется на двустороннюю стрелку (`cursor-nav__icon--drag`).
- Подписей у кейсов нет осознанно — убраны, чтобы не отвлекать от картинки.

Если будешь трогать слайдер — не меняй `flex-shrink`/ширину `.cases__item` не глядя: слайды намеренно `flex: none`, иначе при добавлении клонов вся раскладка начинает сжиматься и по краям появляются пустоты (уже наступали на эти грабли один раз). И не возвращай CSS-transition на transform трека — вся модель построена на том, что transform каждый кадр пишет JS; транзишен поверх даст двойную анимацию и дёргание.

## Медиа кейсов — картинки и видео

Картинки кейсов не финальные и будут меняться/добавляться, часть со временем заменится на короткие зацикленные видео (3–5 сек). Это то, что нужно знать при добавлении новых.

### `loading`/`fetchpriority` у картинок — по факту видимости, не по порядку

Слайдер стартует не с первого кейса, а с центрального по индексу (`startRealIndex` в `app.js`), и его левый край всегда прижат к текстовой колонке (peek — только справа, слева ничего не видно, см. «Слайдер кейсов» выше). Значит **в первом кадре реально видны только активный слайд и небольшой правый peek следующего** — все остальные кейсы не на экране вообще, и грузить их eager-ом бессмысленно.

При добавлении/замене кейса выставляй атрибуты по этому принципу:
- активный на старте слайд (сейчас это кейс-3) — `loading="eager" fetchpriority="high"` (это LCP-кандидат);
- слайд сразу за ним (peek справа, сейчас кейс-4) — `loading="eager"` без `fetchpriority`;
- все остальные — `loading="lazy"`;
- `decoding="async"` — на всех, это не блокирует ничего, просто не отдаёт декодинг главному потоку.

Раньше это было расставлено по порядку в массиве (первые три — eager), что было верно, пока слайдер центрировал активный кейс с peek на обе стороны. После перехода на left-aligned-разметку (только правый peek) разметка не была пересчитана — из-за чего на мобильном грузились eager-ом невидимые кейсы, а видимый peek-кейс — lazy. Если будешь менять логику центрирования/peek ещё раз — пересчитай и эти атрибуты, они привязаны к тому, что визуально видно на старте.

### Почему на мобиле раньше дёргалось: шрифт, не сами картинки

Профилирование на throttled-мобильной сети (Slow 4G, Playwright + CDP) показало: все картинки докачивались за ~800мс, а вариативный шрифт (772 KB, `font-display: swap`) — за 5+ секунд. Всё это время текст рендерился на фолбэке, а когда шрифт наконец приходил — метрики менялись и вся страница (она центрирована по вертикали через `.page { justify-content: center }`) видимо дёргалась разом, слайдер в том числе. Subsetting шрифта (см. «Шрифт» выше) убрал это: тот же прогон стал укладываться в ~850мс целиком. Если жалоба на «дёргание при загрузке» повторится — в первую очередь профилируй сеть заново, не бросайся чинить слайдер.

### Как правильно сжимать видео кейсов

Видео — только фон-декорация внутри карточки кейса, не смысловой контент, поэтому оптимизируем агрессивно. Рецепт (нужен `ffmpeg`):

```bash
ffmpeg -i input.mp4 \
  -an \
  -vf "scale='min(1200,iw)':-2" \
  -c:v libx264 -profile:v high -preset slow -crf 28 -pix_fmt yuv420p \
  -movflags +faststart \
  assets/case-N.mp4
```

Что делает каждый флаг и почему:
- `-an` — убирает аудиодорожку. Видео беззвучное и автоплейное, звук не нужен, а пустая аудиодорожка всё равно занимает место.
- `-vf "scale='min(1200,iw)':-2"` — не даунскейлит, если видео уже уже 1200px, но обрезает всё, что шире. Кейсы показываются максимум на ширине `--content-width` (600px), 1200px с запасом покрывает retina (2x) — грузить видео шире не имеет смысла, ширина экрана его всё равно обрежет.
- `-preset slow` — на файлах в 3-5 секунд лишние секунды кодирования ничего не стоят, а сжимает заметно плотнее, чем `medium`/`fast`.
- `-crf 28` — «золотая середина» для фонового веб-видео (18-20 — почти без потерь, для архива; 23 — стандарт YouTube; 28 — для декоративного контента без потери видимого качества, но заметно легче). Проверено на реальном ассете — 643 KB → 257 KB (-60%) без видимой разницы в кадре.
- `-movflags +faststart` — переносит moov-атом в начало файла, чтобы видео начинало проигрываться сразу по мере скачивания, а не ждало полной закачки.
- `-pix_fmt yuv420p` — гарантирует, что видео проигрывается везде (некоторые энкодеры по умолчанию используют форматы, которые не поддерживает Safari/iOS).

Целевые ориентиры: 3-5 секунд, до ~1200px по длинной стороне, итоговый файл в районе 150-350 KB (если сильно больше — увеличивай `-crf`, не режь длительность).

Разметка для видео-слайда — та же карточка, просто `<video>` вместо `<img>` (`.case__frame`/`.case__image` — классовые селекторы, не завязаны на тег, `object-fit: cover` работает одинаково):

```html
<li class="cases__item">
  <span class="case__frame">
    <video class="case__image" src="assets/case-N.mp4" autoplay muted loop playsinline></video>
  </span>
</li>
```

Обязательно все четыре атрибута: `muted` — без него `autoplay` заблокируют почти все браузеры; `playsinline` — без него iOS раскроет видео на весь экран вместо инлайна в карточке.

**Не реализовано, но стоит учитывать, когда видео-слайды реально появятся:** сейчас `app.js` ничего не знает о `<video>` — он не ставит на паузу видео на неактивных/склонированных слайдах. Если на экране одновременно окажется несколько автоплеящихся видео (например, склонированные буферные слайды), это будет тратить CPU/батарею впустую. Когда будешь добавлять первый видео-кейс — стоит добавить в `app.js` логику вида «play только у активного слайда, остальные paused», по аналогии с тем, как уже управляется `is-active` у иконок курсора.

## Чек-лист перед правками

- Не заводи вторую типографику/колонку — расширяй существующие переменные в `:root`.
- Новую анимацию — через призму скилла `emil-design-eng` (нужна ли вообще, какой easing, какая длительность).
- Анимируй `transform`/`opacity`/`filter`, не `width`/`height`/`top`/`left`.
- Проверяй и десктопный, и тач-сценарий слайдера отдельно — у них разные обработчики.
- Проверяй `prefers-reduced-motion` и мобильную ширину (`max-width: 37.5rem` — единственный брейкпоинт в проекте).
- Новую картинку/видео кейса — сначала через рецепт из «Медиа кейсов» (сжатие, `loading`/`fetchpriority` по реальной видимости), а потом уже в разметку.
- Деплоить не нужно ничего руками — просто `git push` в `main`.
